## 1. Overview

In ScaleChat, "1-on-1" means a private two-party chat between two users **who already have each other in their contacts**. Phone numbers ARE visible to both parties — unlike Super Groups, the disintermediation feature does not apply here. The 1-on-1 surface is the WhatsApp-equivalent direct-message primitive.

**Why this ships before Super Groups.** It is the foundational primitive Super Groups sit on top of (same message bubbles, same input bar, same thread metadata, same repository interface). It is the lowest-risk asset to put in front of Indian brokers/agents for validation. Building it first lets us settle theme, navigation, message components, repository interface, and India-locale handling once — Super Groups inherit the foundation.

## 2. Goals & non-goals

### Goals
- A fully clickable, dark-mode chat experience handling **text + voice notes + contact profile** against an in-memory mock store.
- Production visual quality matching Figma at 392×852 (the design frame size).
- India-locale defaults: `+91`, `HH:mm` + `Yesterday`/`DD/MM/YY`, `en-IN`.
- A clean repository interface that the eventual backend implements without screen churn.

### Non-goals (deferred to later BRDs)
- Real backend, push notifications, end-to-end encryption (mocked as a label only).
- **Voice / video calls** — buttons are visible in Figma but tap → "Coming soon" sheet. Calls are confirmed **free for all users**; provider (recommendation: Agora) + Day-1-vs-later timing belong in the Calls BRD.
- **Status / stories row** at the top of the chat list (visually present as a placeholder, not functional in this slice).
- **Super Group** screens (separate BRD).
- **Account setup / onboarding** (separate BRD).
- **Premium upgrade flow** (separate BRD; only the *gated visibility* of "Add in Super Group" lives here).
- **Settings / Profile screen**.
- **Message retention / disappearing messages** — proposed default: no auto-expiry for 1-on-1 (matches WhatsApp).

### Newly in scope (founder confirmed 2026-05-23)
- **Device-contacts sync** via `expo-contacts` — read the address book and surface ScaleChat-using contacts in the New Chat / Add Contact flow. Permission prompt is deferred until the user first taps **Add Contact** (not at cold start).
- **Voice-note waveform pre-computed on send** — ~80 normalized peaks (~320 bytes) persisted on `Message.voice.waveform` so playback is flicker-free.

## 4. User stories

User is **Mohit** (the name in the Figma greeting "Hi, Mohit"). Each story names the surface (§3 row) it satisfies.

1. **(3.1)** As Mohit, I see all my 1-on-1 chats sorted by recency, with avatar, name, last-message preview, timestamp, and unread badge.
2. **(3.1)** I can filter chats by **All / Unread / Favourites / Group** using the pill chips above the list, without leaving the screen.
3. **(3.1)** I can open the Filter Chats sheet for finer filters (archived, hidden, etc.).
4. **(3.1)** I can search chats by name or last-message text via the search bar.
5. **(3.5)** I can long-press a chat row to enter multi-select mode and bulk-clear or bulk-mark-read multiple chats.
6. **(3.1)** I can tap the `+` button to open the New Chat dropdown with: **New Chat**, **Create Group**, **Add Contact**, **Create Super Group**. Only "New Chat" and "Add Contact" are wired in this slice; the others route to a "Coming soon" placeholder.
7. **(3.2)** I can tap a chat row to open the conversation; messages are grouped by day with a date separator ("Yesterday", "DD/MM/YY").
8. **(3.2)** I can send a text message; it appears as a sent bubble immediately with status `sending` → `sent` → `delivered` → `read` (mocked timeline).
9. **(3.2)** I can long-press the mic icon to record a voice note; the input bar morphs to a recording state with timer; release to send.
10. **(3.2)** I can play received voice notes inline with a waveform/scrubber and duration label (`0:39`).
11. **(3.2)** I can long-press **my own** message → **Reply**, **Select**, **Delete**.
12. **(3.2)** I can long-press **the other person's** message → **Reply**, **Report**, **Select**. (No "Hide contact" in 1-on-1; that action only exists in Super Group contexts.)
13. **(3.2)** I can attach an image or document via the attachment button in the input bar (file is added to the message; mock stores a local URI).
14. **(3.3 / 3.4)** I can tap the conversation header (avatar/name) to open the **Contact Profile**.
15. **(3.3)** From the contact profile I see name, phone `+91 8976543211`, Voice Call / Video Call CTAs, options (Media Links & Docs, Chat Theme, Encryption, Contact Details), Common Groups.
16. **(3.4)** **(Premium only)** I see an additional tooltip with **"Add in or Create Group"** and **"Add in Super Group"**. The tooltip is hidden when `currentUser.isPremium === false`. Tap → routes to a placeholder Super Group picker.
17. **(3.6)** From the conversation header overflow I can open **Clear Chat** (with confirm), **Chat Theme** picker (stub), or **Export Chat** (stub → "Coming soon").
18. **(3.7)** From the chat list overflow I can run **Clear my chats** or **Clear all chats** with a destructive confirm.
19. **(3.2 / 3.3)** Tap the **Voice Call** / **Video Call** buttons in the chat header or contact profile → opens "Coming soon" sheet for this slice. Calls are a **free** feature for all users (not premium-gated) and will be implemented in the dedicated Calls BRD. Provider recommendation: **Agora** (`ap-south-1` low latency, generous free tier suits India).
20. **(3.1, first launch)** On first launch I am prompted to grant **device contacts** permission (deferred until I tap **Add Contact** for the first time, not at cold start). The app reads my address book and surfaces ScaleChat users I already know.

## 6. Data model (`my-app/src/features/chat/types.ts`)

Designed to mirror what `packages/shared` will look like when the backend lands. Fields the mock doesn't use today are marked optional and exist so the eventual swap doesn't change UI code.

```ts
export type Contact = {
  id: string;
  fullName: string;
  phoneE164: string;       // +91XXXXXXXXXX
  avatarUri?: string;
  isFavourite: boolean;
  isMuted: boolean;
  isHidden: boolean;       // from "Hide [contact]" action
};

export type Message = {
  id: string;
  threadId: string;
  authorId: string;        // self id or contact.id
  createdAt: string;       // ISO; renders as HH:mm + relative day
  sequenceNumber?: string; // BigInt-as-string — populated once backend lands
  kind: 'text' | 'voice';
  text?: string;
  voice?: {
    uri: string;
    durationSec: number;
    waveform?: number[];   // pre-computed peaks for the scrubber
  };
  status: 'sending' | 'sent' | 'delivered' | 'read';
  clientMessageId?: string; // idempotency key matching backend contract
  replyToMessageId?: string;
};

export type Thread = {
  id: string;
  contactId: string;       // 1-on-1 → exactly one contact
  lastMessage?: Message;
  unreadCount: number;
  isArchived: boolean;
  isPinned: boolean;
  lastSeenSequence?: string; // BigInt-as-string — will live in MMKV
};

export type ChatFilter = 'all' | 'unread' | 'favourites' | 'group';

export type CurrentUser = {
  id: string;
  fullName: string;
  phoneE164: string;
  isPremium: boolean;      // toggleable in dev to demo subscribed vs unsubscribed
};
```

`sequenceNumber`, `clientMessageId`, `lastSeenSequence` are present from day 1 even though mocks don't use them — they encode the future backend contract so the swap is trivial.

## 7. Repository interface (the seam for the future backend)

```ts
export interface MessagesRepository {
  listThreads(filter: ChatFilter): Promise<Thread[]>;
  getMessages(threadId: string, opts?: { beforeSeq?: string; limit?: number }): Promise<Message[]>;
  sendText(threadId: string, text: string, clientMessageId: string, replyToMessageId?: string): Promise<Message>;
  sendVoice(threadId: string, voice: NonNullable<Message['voice']>, clientMessageId: string): Promise<Message>;
  markRead(threadId: string, throughSeq: string): Promise<void>;
  clearChat(threadId: string): Promise<void>;
  resumeSession(threadId: string, lastSeenSequence?: string): Promise<Message[]>;
}

export interface ContactsRepository {
  list(): Promise<Contact[]>;
  get(id: string): Promise<Contact>;
  add(contact: Omit<Contact, 'id'>): Promise<Contact>;
  setFavourite(id: string, value: boolean): Promise<void>;
  hide(id: string): Promise<void>;
}
```

**Mock implementation** lives in `my-app/src/features/chat/data/`:
- `mock-messages-repository.ts` — in-memory store with `Promise<T>` returns (async-shaped so the swap is type-compatible).
- `mock-contacts-repository.ts` — seeded with Indian names from Figma (Naman Singh, Megha Ahuja, Anand Gupta, Manaj, Right Tanay) + 4–5 more (suggestions: Aanya Sharma, Vikram Patel, Riya Kapoor, Arjun Mehta, Saanvi Iyer).
- `seed-threads.ts` — generates realistic message histories: mix of text + voice, "Yesterday" + today timestamps, varied unread counts.
- `assets/audio/sample-1.mp3` … `sample-4.mp3` — short clips bundled for voice-note playback.

`resumeSession()` is a no-op in mocks (returns `[]`) but the method exists so screen code can call it unconditionally.

### Typography
- Header name: 16px, semibold
- Message text: 15px, regular
- Timestamp: 11px, regular, muted
- Unread badge: 11px, semibold, dark on lime

### Motion
- Chat row press → background flash to `surfaceElevated` (120ms).
- Send message → bubble scale 0.95 → 1 (180ms) + haptic light.
- Long-press message → bubble lift (4dp shadow) + scale 1.04 + haptic medium → reveals action sheet.
- Voice-note play → waveform peaks animate from muted to `accentLime` left-to-right with playback position.

### Premium gate (3.4)
- The **Premium SKU** in ScaleChat is the **Super Group admin subscription** (per pitch: "Admin pays when he creates the super group"). It is the *only* paid product.
- In this 1-on-1 slice, Premium gates exactly one affordance: the **"Add in Super Group"** tooltip on Contact Profile **only renders** when `currentUser.isPremium === true`.
- All other 1-on-1 features (text, voice notes, voice/video calls, themes, export, contact sync) are **free for everyone**.
- Dev toggle: a hidden `__premium=true` query param on the profile route flips the flag in mock state so we can demo both views without rebuilding.

### Voice-note waveform
- **Pre-computed on send.** When the user finishes recording, we sample peaks (typically 60–80 normalized floats) and persist them on the `Message.voice.waveform` field. Playback uses these peaks directly — no flicker, instant scrubber paint.
- Storage cost: ~80 floats × 4 bytes ≈ 320 bytes per voice note. Negligible.
- Library: `react-native-audio-api` waveform extraction OR offline FFT via `expo-av` recording metadata; pick during implementation against v56 docs.

## 9. Edge cases

| Case | Behavior |
|---|---|
| Empty chat list | Centered illustration + "Start a new chat" CTA → opens New Chat dropdown |
| Long names | Single-line ellipsis truncate at row width |
| Long last-message preview | Single-line ellipsis truncate |
| Unread count > 99 | Render `99+` |
| Voice note > 5 min | UI caps display at `M:SS / 5:00`; backend will enforce later |
| Voice note recording interrupted (phone call) | Save partial recording; show "tap to keep / discard" prompt |
| Network offline | Mock world ignores; future: queue sends and replay on reconnect (the `clientMessageId` field is the idempotency key) |
| Contacts permission denied | New Chat dropdown still shows "Add Contact" (manual entry) and ScaleChat users from prior sessions; surface a "Grant access" pill at top of contact picker so user can reconsider |
| Multi-select > 5 chats | Action bar shows "5+" count; bulk actions still operate on all selected |

## 12. Decisions & remaining open questions

### Resolved (2026-05-23)
- **Voice / Video calling** — **FREE for all users** (not premium-gated). Provider not yet locked; recommendation is **Agora** for India + Fly Mumbai region. Timing (Day-1 vs post-Super-Groups) deferred to Calls BRD.
- **"Hide contact" in 1-on-1** — **NOT in scope.** Removed from user stories §4.12 and edge cases. The `Contact.isHidden` field stays on the type because the Super Group BRD may use it; in 1-on-1 surfaces the value is always read as `false`.
- **Premium SKU scope** — **strictly the Super Group admin subscription.** It gates only the "Add in Super Group" affordance on Contact Profile (3.4). All 1-on-1 features (text, voice notes, voice/video calls, themes, export, contact sync) are free for everyone.
- **Contact sync** — **YES, device contacts.** Read the device address book and suggest matches. Permission prompt is deferred until the user taps **Add Contact** the first time (not at cold start). Library: `expo-contacts` (Expo SDK 56).
- **Voice-note waveform** — **pre-computed on send.** ~80 normalized peaks (~320 bytes) persisted on `Message.voice.waveform`. Better playback UX, negligible storage cost.

### Still open
- **Calls provider** — Agora vs Twilio vs Stream. Recommend Agora; defer to Calls BRD.
- **Message retention** — proposed default: **no auto-expiry for 1-on-1** (matches WhatsApp). The backend has a `messageExpiry` BullMQ worker, but it stays unused in 1-on-1 unless the user toggles per-chat "disappearing messages" (post-MVP). Super Group retention is a separate decision in the Super Group BRD.
- **Custom 5-slot tab bar implementation** — `expo-router/unstable-native-tabs` doesn't natively support the center-FAB layout; decision (custom Tabs layout vs floating FAB over NativeTabs) belongs in the next implementation plan, not this BRD.
